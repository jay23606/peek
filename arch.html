The Zero-Server Stack

How a static page, a managed database, and a wall replaced the web server — and where that trade still doesn’t hold.

Topic  Serverless web application architecture Audience  Engineers & technical decision-makers Revised  July 2026
Abstract

A large and very common class of web application no longer needs a web server. When the frontend is a set of static files, the database is fully managed, and access control is enforced inside the data layer rather than in application code, the browser can talk almost directly to the database — with a security “wall” in between deciding what each request is allowed to see. This paper describes that pattern, which we call the zero-server stack; explains the mechanism that makes it safe; weighs its genuine advantages against its real limitations; and surveys ten managed backends that implement it. The conclusion is deliberately modest: the pattern is not the end of servers, but a sensible new default for applications that are mostly create-read-update-delete with accounts and live updates. It is, in the words of the source it draws on, “a great default — not a religion.”

01The shift, in one picture

The same job, half the plumbing

For most of the web’s history, serving an application meant running a fleet of machines. A visitor’s request travelled through a content-delivery network to a load balancer, which spread traffic across two or more web servers, which in turn queried a database. Each layer had to be provisioned, patched, scaled, and watched. The web server, in particular, did a great deal of quiet work: it held secrets, checked who you were, decided what you could see, ran scheduled jobs, received webhooks, and wrote logs.

The zero-server stack asks a blunt question about that machinery: what if we deleted the web server and let the browser talk to the database directly? The answer, for a surprising range of applications, is that everything still works — provided each job the server used to do is reassigned to a component that can do it more cheaply.

Three things had to mature before this became reasonable. Managed databases now ship with authentication, file storage, real-time subscriptions, and a scheduler built in. Databases can enforce access rules per row, so a client can be handed a raw query interface safely. And static hosting on a global CDN — for example, GitHub Pages — became free and effortless. Put together, they let a small team ship a production application with, quite literally, no server to run.

02How the pattern works

Reassigning the server’s jobs

The pattern is best understood as a redistribution of responsibilities. Nothing the server did disappears; each duty simply moves to a component better suited to it. The frontend ships as static files. The database does the data work. Two new helpers — the wall (row-level security) and edge functions (small serverless functions that run only on demand) — absorb the rest.

Edge functions: hired help that clocks in only when needed

The browser cannot be trusted with secrets. Anything shipped in JavaScript is readable by anyone, so a payment provider’s secret key or an email service’s credentials can never sit in client code. Edge functions solve this: they are small pieces of server-side code that spin up on demand, do the one privileged thing, and disappear. They are also where webhooks land — when a payment processor needs to “phone home” after a charge, it calls a public edge-function URL, which verifies the caller before acting. You pay for the milliseconds they run, not for a machine that idles.

The built-ins: auth, real-time, cron, storage, logs

The remaining duties are simply features of the managed backend. Authentication is built in — email-and-password, magic links, and “Sign in with Google” without writing an auth server. Real-time is native: when a row changes, every browser subscribed to it hears about the change, which is enough to build chat, live scores, or presence indicators. A built-in scheduler lets the database “wake itself up on a timer” for cleanup, reports, and reminders. File storage sits alongside the data, and logs are just another table — searchable with the same tools as everything else.

03The wall: security by row, not by secrecy

You don’t hide the queries — you guard the rows

The obvious objection to letting a browser talk to a database is security. If the client can send queries, surely it can read anything? The answer is the single most important idea in the whole pattern, and it inverts the usual instinct. You do not try to hide what queries are possible. You assume the attacker can see everything about the shape of your data, and you guard the rows themselves.

You don’t hide the queries. You guard the rows.The Zero-Server Stack

Because the browser talks straight to the database, a number of things ship in plain view, and the design treats that as fine:

Public by design

  • The public “anon” key in the JavaScript
  • Your table and column names
  • The structure of every query
  • Any query the rules actually permit

Never exposed

  • The master key (service role)
  • API secrets for payments and email
  • Optionally, the query surface itself — locked to named actions

The guarding is done by Row-Level Security (RLS): rules attached to each table that the database evaluates on every request. A policy is a small condition. Here, a user may select a post only when it is their own:

create policy "own posts" on posts
  for select using ( auth.uid() = user_id );

The elegance is in what happens to an attacker. Their query still runs — the database does not reject it with an error that reveals structure — it simply “comes back with only their own rows.” The public key opens the door; the wall decides what leaves with them.

The mistake that burns people

Turn the wall on for every table. A table with the rules switched off is a fully public table. RLS is opt-in per table, so a single forgotten toggle exposes everything in it — there is no second layer of application code behind it to catch the miss. This is the pattern’s sharpest edge, and it is discussed again among the limitations below.

04Where it wins, where it hurts

A balanced ledger

The advantages are real and mostly economic; the limitations are real and mostly architectural. An honest evaluation keeps both columns in view.

Where it wins

  • Cost approaches zero. A free CDN plus a free or near-free database tier replaces a monthly infrastructure bill.
  • No operations burden. Nothing to patch, scale, or wake at 3 a.m. The provider owns uptime.
  • Speed to market. Auth, storage, real-time, and cron arrive on day one instead of being built.
  • Fewer moving parts. No load balancer, no server fleet, no deploy pipeline for a backend that no longer exists.
  • Security in one auditable place. Access rules live with the data, not scattered across endpoints.
  • Real-time is free. Live updates are a native feature rather than a second system to run.

Where it hurts

  • RLS is a single point of failure. One un-walled table leaks everything in it; the safety net of server-side code is gone.
  • The query surface is public. Schema and query shape are visible, inviting probing; some teams lock the client to named actions to compensate.
  • Vendor lock-in. Auth, rules, and functions are provider-shaped. Open-source, self-hostable options soften this; closed ones do not.
  • Cold starts & idle sleep. Free database tiers may pause when idle, and edge functions can lag on first call.
  • Cost cliffs at scale. Generous free tiers give way to usage-based pricing; egress and function volume can climb quickly.
  • Heavy or long-running compute doesn’t fit. Edge functions are short-lived; there is no always-on process to lean on.

The three jobs that still want a real server

The source document is candid about its own boundaries, naming three cases where a traditional server remains the right tool. This paper endorses that list and treats it as the core “when not to”:

  1. Fresh-per-visit page rendering. When each request must assemble bespoke HTML on the server — common in large content and commerce platforms — pre-built static files are not enough.
  2. Always-on background processes. Long-running, stateful work — a queue consumer, a game server, a persistent socket hub — has no home in functions that clock in and out.
  3. Network-level gatekeeping. Custom certificates, IP allow-lists, and specialised traffic rules need infrastructure you control at the network layer.

To these three, add the practical caveats above — the RLS footgun, lock-in, idle behaviour, and cost cliffs — and the picture is complete: a strong default for a specific shape of app, with clearly marked edges.

Reading the field

The PostgreSQL family — Supabase and Nhost. Both put a real relational database in front of the browser, with row-level security as the wall. Supabase exposes REST and GraphQL and adds full-text and vector search; Nhost leans GraphQL-first. These are the most direct embodiment of the pattern described here.

The self-host-first pair — PocketBase and Directus. PocketBase is a single executable over SQLite that never sleeps and stores as much as your disk allows — ideal for small, self-run apps. Directus wraps an existing SQL database, making it attractive when the data already exists.

The reactive outlier — Convex. Rather than exposing a query language, Convex runs typed TypeScript functions and re-runs them reactively when data changes. Its access model is function-shaped rather than row-shaped (hence the partial marks on auth and record-level access), a different but coherent take on the same problem.

The document store — Appwrite. A mature open-source, self-hostable backend built around NoSQL documents, strong on auth and storage, lighter on search.

The incumbents — Firebase, Amplify, Back4App, Backendless. Firebase (Google) is the most battle-tested and never sleeps, but is closed and cannot be self-hosted, and its search is weak. Amplify sits on AWS DynamoDB with the largest free database allotment (25 GB) but is the only option here that requires a credit card. Back4App (a hosted Parse) and Backendless (low-code) round out the field.

Two columns deserve special attention when choosing. Idle sleep splits the field: Convex, PocketBase, Firebase, Amplify, Backendless, and Directus stay awake, while Supabase, Appwrite, and Nhost may pause a free project until the next request — a first-visit delay that matters for low-traffic apps. And open-source with self-hosting — true for Supabase, Appwrite, Nhost, Convex, PocketBase, Back4App, and Directus — is the practical antidote to lock-in: if the hosted tier disappoints, the same stack can run on your own machine.

The biggest saving isn’t on the invoice — it’s the person-hours that never get spent patching, scaling, and firefighting infrastructure.The Zero-Server Stack

This is the argument’s strongest form. For a small team, the dollar difference is pleasant; the disappearance of the on-call rotation, the patch cycle, and the scaling scramble is transformative. It is worth remembering the other side of the curve, though: usage-based pricing means a sudden success — a viral moment, a data-heavy feature — can move a project off the free tier quickly, and egress and function volume are the usual culprits. The model saves the most for applications that are modest in scale or predictable in growth.

07A decision guide

Choosing, in one page

Reach for zero-server when…

  • The app is mostly CRUD + accounts + live updates
  • You want zero upfront infrastructure cost
  • There is no dedicated operations staff
  • Speed to market outranks bespoke control
  • Traffic is modest or predictable

Keep (or add) a server when…

  • Pages must be server-rendered fresh per visit
  • You run heavy or always-on background work
  • You need network-level control — certs, IP rules
  • A deliberate service layer is part of the design
  • Compliance demands data residency you control

A common and sensible outcome is hybrid: run the zero-server stack as the default for the bulk of the application, and stand up a single small server only for the one job — a rendering path, a worker, a gateway — that genuinely needs it. The pattern does not have to be all-or-nothing to pay for itself.

08In practice: Peek and Mayfly

Two case studies — and where each pushes further

Peek — the app this document ships alongside — is a near-literal instance of the zero-server stack: one static page on GitHub Pages talking to a single Supabase project, with row-level security as the only gatekeeper. There is no application server and, tellingly, not even an edge function — the app holds no server-side secret, so it never needs one. It also does one thing the base pattern doesn’t ask for, and that twist is the most interesting part.

Your full-resolution photos never touch a server.Peek README

The wall, doing real work

The paper’s one-line policy grows up here. A post is visible only if it is yours, its author is public, or you are an accepted follower — private accounts and all — and that rule lives entirely in the database:

-- yours, or a public author, or an accepted follower
create policy "posts_select" on posts for select using (
  auth.uid() = user_id
  or exists (select 1 from profiles p
             where p.id = posts.user_id and p.is_private = false)
  or exists (select 1 from follows f
             where f.following_id = posts.user_id
               and f.follower_id = auth.uid() and f.status = 'accepted')
);

The browser may send whatever query it likes; the wall returns only the rows the viewer is entitled to. Peek even stretches RLS onto real-time itself — private group-chat channels named group:<id> are made member-only by a policy on the Realtime message table, so a stranger can’t subscribe to a group they aren’t in.

Pushing past the pattern: photos that never touch a server

The stock pattern would drop uploaded images into the backend’s file storage. Peek refuses. Only a ~32-pixel blurred preview — a tiny base64 string — is stored in Postgres; the full-resolution image lives only in the author’s browser (IndexedDB) and streams browser-to-browser over a WebRTC data channel when a viewer is online. Even the WebRTC signaling rides Supabase Realtime, so no server ever sits in the media path.

This engineers out of existence the paper’s sharpest cost worry — storage and egress at scale. Because the database only ever holds text and thumbnails, the whole app sits inside Supabase’s free 500 MB tier essentially forever, no matter how many megapixels its users shoot.

The bill for that twist

The saving is paid for in exactly the currency the limitations named. Peer-to-peer is live-only: if the author is offline the post stays blurred, and direct messages and calls need both people present and keep no history. And with only public STUN and no TURN relay, the peer connections “often fail on mobile cellular data” behind carrier-grade NAT. The Supabase-backed half — feed, likes, comments, profiles, discovery — works everywhere; the peer-to-peer half is precisely where one small piece of real infrastructure (a TURN server) would be the honest fix.

Mayfly: the same stack, with a delivery ladder

Mayfly uses the same basic ingredients — a static PWA on GitHub Pages, Supabase Auth, Postgres, Realtime, Storage, and row-level policies — but asks a harder question: what happens when the product is the message itself? Its answer is not one transport but a ladder. A direct Snap prefers a browser-to-browser WebRTC transfer while both people are online; when the recipient is offline, the media is encrypted to that recipient’s published device key before an opaque relay blob is placed in Storage. The server can route the ciphertext, but cannot render the Snap.

Pattern pieceIn Mayfly
Static frontendPlain ES modules, no build step, and an installable PWA served by GitHub Pages.

That design makes the zero-server boundary more visible. A managed database is excellent at identity, authorization, notification metadata, and an encrypted mailbox. It is not a substitute for a continuously available peer: an online-only author cannot serve a live Story after closing the app, and group media cannot reach members who are not present. Those are product choices, not implementation accidents, and Mayfly makes them legible.

The next additions worth making explicit

Mayfly also supplies a practical roadmap for the point where a nearly serverless app earns a few carefully bounded helpers. None needs to become a conventional application server; each exists because it performs one job the browser and RLS layer cannot reliably perform alone.

Lifecycle jobs24-hour products need predictable cleanup of expired metadata, relay blobs, and device copies without relying on a particular browser to wake up.Scheduled database jobs or a narrowly scoped cleanup function.
Connection observabilityPeer-to-peer failures otherwise look like vague “connecting” states and are hard to distinguish from camera, permission, or NAT problems.Privacy-preserving client telemetry: permission failure, ICE state, relay use, call duration, and error class — never message or media contents.
The useful boundary

These additions are not a retreat from the zero-server stack. They are the discipline that makes it credible: keep identity, data, and ordinary authorization in the managed platform; add a trusted component only for connectivity, push delivery, abuse handling, recovery, scheduled cleanup, or secrets.

Together, the two apps make the thesis concrete. Peek follows the zero-server default almost to the letter — no server, no edge function, not even file storage. Mayfly keeps that same center of gravity while admitting an encrypted Storage fallback for offline Snaps. In both cases, the rule is the same: add a server-shaped component only where it does one clearly bounded job the browser and the wall cannot do honestly.

09Conclusion

A great default, not a religion

“You can delete the server” is a provocation, and like most good provocations it is true within limits worth stating plainly. When an application is built from static files, a managed database, and a wall of row-level rules — with edge functions clocking in only for the privileged work — the traditional web server has no remaining job, and removing it removes an entire category of cost and toil. That is a genuine and welcome shift for a large slice of the applications people actually build.

It is not universal. Fresh-per-visit rendering, always-on processes, and network-level control still want real infrastructure; lock-in, the RLS footgun, idle sleep, and cost cliffs are real edges to respect. The honest framing is the one the source itself lands on, and the one this paper adopts: treat the zero-server stack as an excellent default for CRUD-with-accounts-and-live-updates, choose an open-source backend where possible to keep the exit open, turn the wall on for every table — and reach for a server only when the work in front of you is one of the few things a server still does best.
